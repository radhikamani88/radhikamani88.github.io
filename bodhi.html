Re-making of Bodhi

An enterprise Operating system testing platform

Role

As an intern, I led the entire project with assistance from my mentor Mo. My mentor had given me sufficient accesss and freedom to pursue this project on my own and she was always available to provide me assistance and reviews.

The challenge

Bodhi is a modular web-based system where red hat's developers, testers and the open source community users collaborate on testing, voting and updating packages.

Upon taking up this project, I quickly wanted to get a sense of the problem space before I plan my approach to solving the problems. With that in mind, I quickly had some discussions with some Bodhi users who were already present that day on that floor and nearby floors.

Here are a few things I heard:

I use bodhi everyday, but I often get confused by what package to work on

Developer-1, Redhat Inc.

I feel I am comfortable using the command line interface. It's easier to use than the UI

Developer-2, Redhat Inc.

I got a sense that developers were not too keen on using the web interface and testers had a hard time navigating the web interface. This was especially true when new testers were introduced to the web UI. The challenge was to increase the Web UI engagement amongst the community.

Existing system

User research

As I sat down to ponder more on the problems being faced by the users, I divided them into 3 main categories for my redesign project.

  • Testers and release engineers working for Redhat who frequently use the web application.
  • Packagers and Developers working for Redhat who sometimes use the web UI as well as the command line interface.
  • Open-source contributors who were not employees of Redhat

The Contextual Inquiry

I interviewed testers, developers, packagers and release engineers and observed their day to day process while they are in office. This way it helped me understand their process flow better and note down any problems they were facing. Based on the interviews, I built an affinity wall to understand their concerns and find any potential use cases for redesign.

    Don't forget the heuristic evaluation!

    Even after collecting all that data from user interviews, there were certain pages for which I did not have enough information. So, I conducted a heuristics evaluation of the website to see what all other problems that website might have and zeroed on the following problems in the UI.

    Here's a summary of the issues.

    Flexibility and efficiency of use
    The primary problem was finding the right package or searching a package to work on. Currently in the system it was not easy to distinguish whether the package belonged to F25, EPEL or F24. Easily guiding the user to find what they were looking was the key to a better update system.
    Aesthetic and minimalist design
    There was way too much information and it was not hierarchically organized in the page which led to making the page appear more cluttered and confusing to use.
    Help users recognize, diagnose, and recover from errors
    No error messages was displayed when the user was about to make mistakes while changing the package contents.
    Help and documentation
    Further, having to search for fedora wiki pages for documentation was both time consuming and cumbersome.

    Survey design

    Once I brainstormed several solutions, I wanted to gather some data on which alternatives might be better. First, I reviewed my ideas with my mentor. My mentor's response was positive on the ideas. We thought it might be better to gather data from real customers on the ideas that I had. Instead of showing customers with all the alternative solutions that I had, I prepared a survey that included some basic questions to understand the audience and a question each for each of the various tasks for which I had evaluated multiple solutions. For each of them, I provided them a view of the current system as well as my proposed solution and asked them a bunch of questions to help me find out how much they preferred the second solution.

    Here is a link to the survey

    The 2 tasks for which I gathered data in this survey were the Download widget and the Bugzilla integration. I chose these two only because they were the only new features that I was proposing we add to the website. Because they were new features they would also require additional back-end integration work and quantifying the benefit of these new features would make it easy for management to prioritize the implementation of these features.     Final design

    Based on the positive feedback from the survey and from my mentor on the design proposals presented as low-fi designs, I went forward and made the hi-fidelity designs for the same. Redhat had recently come up with a new style guide called Patternfly for enterprise applications. Since I was re-designing Bodhi from the ground up I decided to use this new style guide. Since, CSS files were already available for patternfly, it turned out to be easier to build the hi-fidelity prototype by just coding it using the Patternfly library

    Home page

    This home page will consists of critical path in need of testing, Security and Normal updates grouped under the release F25, EPEL to allow for more clearer representation of the information

    Statistics

    Will have the overall metrics and weekly top testers information

    Update details

    This page will show the package details, Bugzilla updates, Download widget, comments and Karma

    What I learned from this experience

    I felt that employees suffer more because most enterprise applications do not provide engaging user experience. This is because of the fact that most of the decisions taken by the software managers are primarily concerned with stability, security and scalability and not usability. Thus, it is not an easy goal to achieve and more often than not there are lot of that a lot of esoteric constraints when designing an enterprise application. So,sometimes it becomes very hard to resist becoming a pushover designer. However, that itself can be taken as a good challenge.

    What I enjoyed most in my internship at Red Hat is that, my users were literally down the hall and I can always meet and emphatize with them. I really enjoyed the informal conversations I had with them. Having such relaxed conservations enabled them to speak freely about their problems and pain points. Well, who doesn't like the company of honest people!
