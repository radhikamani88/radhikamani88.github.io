Making of Portfolio website

Letting the work speak for itself

I wanted to make a portfolio website to showcase all my work. This was a dauting task at first for me because I had never built a website before and I was not sure what I should put in website. Soon, a thought occured to me — why not approach the portfolio website design as a UX problem?

Attacking it as a UX problem

Upon inspecting it as a UX problem, I realized that I first needed to define a success criteria (goal) for my portfolio. In turn that led me to ask — Who is the target audience?

I kept the target audience was prospective employers. I then asked myself — what would prospective employers like to see in my portfolio?

Research

It would have been awesome if I could directly ask the prospective employers what they expected out of a portfolio. Since, that was not possible, I did a bunch of research of my own:

  • Talked to seniors at my university and visited their portfolio websites
  • Looked at portfolios of UX designers employed already
  • Read various internet articles on building a portfolio

Goals

Based on above research I set the following design goals.

Let work speak for itself
I didn't want this website to be the primary source of interest for the reader. I wanted reader to be more focused on my project work
Reader can quickly assess work quality
I was aware that readers have to read hundreds of portfolios and are short on time. Help them help me.
Reader is guided through my work
Since I won't be present to walk through the portfolio the reader, the website should be able to guide reader through the portfolio
Reader can read on any device type
This meant the website need to be responsive at-least and being adaptive to device type would be a very nice thing to have

Explorations & decisions

KISS principle. Occam's razor

No fancy animations and Javascript libraries that distract or conflict with animation present in your actual work. Keep website design simple and let your real work shine through. When it doubt choose the simpler option.

Home page IA

Above design goals led me to research and think deeply about the Information Architecture (IA) of this website. I sketched quickly various alternatives for the home page IA, evaluated pros-cons and picked one alternative.


Different solutions evaluated for home page IA

Grid of projects


Discarded because:

  • didn't meet the criteria of allowing me to guide the user through the portfolio.
  • Presented with a grid of options, all the projects in a row are presented at an equal accessibility and importance level and the reader would be inclined to click any one of them.

Vertical list of project cards


The projects presented top-down in a list implicitly rank the projects and the reader is inclined to click and read the first one. However, I discarded it because:

  • Each item's primary content is an image and doesn't allow me to provide some summary information about each project which will allow reader to quickly assess impact of reader.
  • An image alone may not give enough clues about the impact of this project and may not interest reader enough to read the project.

Vertical list of projects w/ blurb and image


Selected because:

  • it allowed me to present information in a prioritized fashion
  • present a short blurb describing purpose, impact, role and technologies involved in this project along with the project image
  • allows readers to quickly scan through the crux of this project.

Typography




the standard typographic scale as an example

I spent a considerable amount of time researching about setting the tyography of the website. This was the first time I had embarked on a deep study of typography and arrangement of textual content on websites. Since, I only had basic education background on this topic, I started researching online and looked around for resources on this topic.

I found the following book very useful and many of my typographic decisions are based on this book: Elements of typographic style applied to the web by Robert Bringhurst. Robert gives a clear set of guidelines for typographic newcomers to follow to get typesetting basics correct.

In implementing the typesetting for this website, I applied the following design choices:

  1. A mixed font-scale is used. The difference between body text and h1 text size is based on the traditional font scale. The difference between h1, h2 and h3 is approximating the perfect-fourth scale.
  2. The entire text is laid out on an imaginary notebook-like canvas where the line-height is based on the base font size of the paragraph text
  3. margin + line-height + padding = N x base font line-height
  4. Regardless of combination of h1, h2, h3 and p elements used the text is aligned properly on the line-height boundaries on this imaginary canvas.
  5. The line width of the main content of the website is set so that approximately at-most 8-15 main words fit per line. This usually comes to around 88-90 characters per line.
  6. The font used is ff-meta-web-pro for main body text and freight-sans-pro for the headlines
  7. The text is flush left aligned
  8. No use of h4, h5 and h6 elements. Content is adjusted if needed
  9. First paragraph does not have a text indentation but all subsequent paragraphs do and the subsequent paragraph continues on the next line in this imaginary notebook.

Edward Tufte - Margin notes, links
and mixing graphics with text


© Edward Tufte

I am a fan of Edward Tufte's books.

Tufte's books follow a specific typographic style and he strongly advocated mixing graphics with text to make text more interesting to read and to aid in explanation of text. I have followed that practice in this website as well. I felt the guidance was very appropriate for portfolio websites.

Edward Tufte also provides a special markup for hyperlinks. The default underlining provided by browsers for hyperlinks is very thick and it draws the link over the descenders of letters in text. Tufte provides CSS code which does better job at underlining links.

Tufte's books make extensive use of margin notes to provide additional information that is useful but not necessary for completeness of main content. I have followed this guideline to provide additional imagery and useful information.

The practices followed by Edward Tufte have been put down in a CSS file as a Github project called Tufte-CSS

Development

Themes and Wix didn't suffice

First I tried using an existing theme from the internet and altering it. However, that was not extensible enough to be able to modify it to meet the new design that I had produced. I ran into the same issue with many themes.

I then turned to Wix. I was initially satisfied with Wix. I really like that the fact that I completely control the layout of the website. Sadly, it was too good to be true.

Wix has the following drawbacks:

  1. Wix websites are not mobile-friendly or responsive by default
  2. Wix does give us a mobile viewer, however, that is severely lacking in features and there is no easy way to check if the website will look good for different device types. Google chrome has better support for testing responsiveness than Wix.

Decided to hand code

I really want to get full control of the layout and be able to reuse my layout specifications across all pages and minimize duplicate work. As existing tools did not suffice, I hand coded it myself.

The website is built using Bootstrap v3. I apply the DRY principle and use SCSS and Jade to that advantage.
